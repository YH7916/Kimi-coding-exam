当前这个项目里，“记忆系统”不要做成一个新 Agent 工具。因为 Phase 3 明确要求 Agent 只有 `readFile(fname)` 一个工具，贸然加 `saveMemory/searchMemory` 会破坏题目边界。

我建议这样做：

**1. 短期记忆：保留现在的 history 机制**

这个已经有了：

- 前端传 `history`
- 后端 `ChatRequest.history`
- 检索时用 `_conversation_query(message, history)`
- LLM prompt 里用 `_history_messages(history)`

这层解决：“那升级条件呢？”、“继续刚才那个 OOM 问题” 这种追问。

**2. 工作记忆：保留每轮 Agent 的 trace/evidence**

也已经有了：

- retrieval candidates
- `readFile` tool calls
- evidence extraction
- final answer trace

这层解决：“你为什么这么回答？”、“你读了哪些 SOP？”。

**3. 长期记忆：只做 optional case memory，不要影响主线**

如果要加，可以做成“历史事故案例库”，不是 Agent 工具。

结构类似：

```json
{
  "id": "mem-001",
  "created_at": "...",
  "incident_type": "backend_oom",
  "symptoms": ["服务重启", "JVM 内存飙升"],
  "actions": ["保存 heap dump", "限流", "回滚版本"],
  "outcome": "恢复服务",
  "source_files": ["sop-001.html"],
  "tags": ["OOM", "P0", "backend"]
}
```

然后检索流程变成：

```text
用户问题 + 近期 history
        |
        |-- v2 SOP hybrid retrieval
        |-- memory case retrieval
        |
     prompt context
        |
Agent 仍然只能 readFile(SOP)
```

关键原则是：**SOP 是权威来源，memory 只能作为“历史参考案例”**。回答里也要标清楚：“根据 SOP ...；历史类似案例中 ...”。

**实现优先级**

1. 现在先不动，当前 history 已经够交付考试。
2. 如果要加亮点，就加一个 `MemoryRepository`，用 SQLite/JSONL 存历史案例。
3. 不给 Agent 新工具，只在 `SearchRuntime.chat()` 里提前检索 memories，然后作为 system context 注入。
4. 加测试：SOP 证据优先、memory 不能覆盖 SOP、敏感信息不入库、history 追问仍然正常。

一句话：**短期记忆用 history，长期记忆做案例库，千万不要把 memory 变成第二个工具；否则会和题目“只有 readFile 一个工具”的约束打架。**
